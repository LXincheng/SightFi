# SightFi 项目规则（Rules）

> 目标：保证“开发快、部署稳、耦合低、可长期维护”。

## 1. 文档与流程规则

| 编号 | 规则             | 说明                                 |
| ---- | ---------------- | ------------------------------------ |
| R-01 | 文档单一目录     | 所有项目文档统一放在 `docs/`         |
| R-02 | 变更先更新计划   | 开发前先核对 `docs/plan.md` 当前阶段 |
| R-03 | 里程碑后更新日志 | 每个里程碑完成后更新 `docs/log.md`   |
| R-04 | 需求变更同步 PRD | 需求范围变化必须同步 `docs/prd.md`   |

## 2. 架构与仓库规则（Monorepo）

| 编号 | 规则               | 说明                                                  |
| ---- | ------------------ | ----------------------------------------------------- |
| A-01 | 前后端同仓         | 使用 Monorepo：`apps/web` + `apps/api` + `packages/*` |
| A-02 | 低耦合优先         | 前后端通过契约通信，不共享运行时状态                  |
| A-03 | 共享仅放类型和工具 | `packages/shared` 仅存 type/const/util，不放业务逻辑  |
| A-04 | BFF 统一出口       | 前端不得直接调用第三方行情/新闻 API                   |
| A-05 | 可替换数据源       | 第三方源必须走 adapter，避免硬绑定单一供应商          |

## 3. 编码与质量规则

| 编号 | 规则                  | 说明                                 |
| ---- | --------------------- | ------------------------------------ |
| C-01 | TypeScript 禁止 `any` | 必须使用明确类型或泛型               |
| C-02 | 异步必须可恢复        | 每个异步流程必须有错误处理与用户反馈 |
| C-03 | Secret 禁止硬编码     | 一律走环境变量与配置层               |
| C-04 | 三处重复即抽象        | 代码重复超过 3 次必须提取            |
| C-05 | 小步提交可回滚        | 单次改动保持可读和可回滚             |

## 4. 数据与 AI 规则

| 编号 | 规则          | 说明                                  |
| ---- | ------------- | ------------------------------------- |
| D-01 | 数据可追溯    | 每条核心数据保存 `source_id` 与时间戳 |
| D-02 | 冲突不覆盖    | 多源冲突时标注“待确认”，不静默覆盖    |
| D-03 | AI 必须证据化 | 结论必须附证据与置信度                |
| D-04 | AI 不得绝对化 | 禁止“保证收益”“必涨必跌”等表达        |
| D-05 | 记忆可删除    | 用户可查看、编辑、清除 AI 记忆        |

## 5. UI 与体验规则

| 编号 | 规则             | 说明                                  |
| ---- | ---------------- | ------------------------------------- |
| U-01 | Figma 为 UI 真源 | 布局与交互优先对齐 Figma 原型         |
| U-02 | 响应式为默认     | 新页面必须先过移动端，再扩展桌面端    |
| U-03 | 主题双模强制     | Day/Dark 必须等价可用，不允许只做反色 |
| U-04 | 信息不过载       | 单屏关键卡片控制在可扫描范围          |
| U-05 | 性能优先         | 动效必须可降级、可关闭                |

## 6. 测试与发布规则

| 编号 | 规则                       | 说明                                |
| ---- | -------------------------- | ----------------------------------- |
| T-01 | 每个模块至少一类自动化测试 | 单测或集成测试至少其一              |
| T-02 | 主分支必须可构建           | CI 失败禁止合并                     |
| T-03 | 发布前回归清单             | 覆盖登录、看板、持仓、新闻、AI 输出 |
| T-04 | 监控与告警先行             | 新增关键链路必须带基础指标和告警    |

## 7. 非目标（避免复杂化）

| 编号 | 不做项                 | 原因                     |
| ---- | ---------------------- | ------------------------ |
| N-01 | 首版不做自动交易       | 合规与风控成本过高       |
| N-02 | 首版不做复杂微服务拆分 | 与“轻量开发部署”目标冲突 |
| N-03 | 首版不做重型 BI 平台   | 先聚焦核心监控与研究闭环 |
