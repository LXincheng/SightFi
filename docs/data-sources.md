# SightFi 第三方数据源清单（Phase 1）

> 目标：明确每个第三方数据源的用途、接入方式、降级策略与是否需要 Key，保证可维护和可替换。

## 1. 总览矩阵

| 数据类型                 | 方案/API                   | 当前状态         | 使用位置                        | 是否需要 Key     | 选择原因                                                  |
| ------------------------ | -------------------------- | ---------------- | ------------------------------- | ---------------- | --------------------------------------------------------- |
| 美股/港股/ETF/重金属行情 | `yahoo-finance2` (Node.js) | 已适配（可开关） | API `GET /api/v1/market/quotes` | 否               | 免配置、接入快、覆盖面广，适合作为 Phase 1 主力轻量行情源 |
| 全球金融新闻             | `Financial Modeling Prep`  | 已适配（可开关） | API `GET /api/v1/news/facts`    | 是               | REST 简单、金融领域结构化较好、免费额度可用于 MVP         |
| 地缘政治/聚合新闻        | `GNews`                    | 已适配（可开关） | API `GET /api/v1/news/facts`    | 是               | 关键词聚合能力强，适合宏观与地缘事件追踪                  |
| 综合新闻补充             | `NewsAPI`                  | 已适配（可开关） | API `GET /api/v1/news/facts`    | 是               | 全球覆盖面稳定，作为新闻备源                              |
| 中国宏观/金十/A 股深度   | `AkShare` + 内部网关       | 规划中           | 后续 `macro` 模块               | 否（对 AkShare） | 数据维度丰富，但需 Python 网关，Phase 1 先不强耦合        |

## 2. Provider 策略

| 模块 | 环境变量               | 取值                                          | 默认值 | 行为                                               |
| ---- | ---------------------- | --------------------------------------------- | ------ | -------------------------------------------------- |
| 行情 | `MARKET_DATA_PROVIDER` | `mock` / `yahoo`                              | `yahoo` | `yahoo` 失败自动降级 `mock`                        |
| 新闻 | `NEWS_DATA_PROVIDER`   | `mock` / `auto` / `fmp` / `gnews` / `newsapi` | `auto` | `auto` 下按 `GNews -> NewsAPI -> FMP -> Yahoo RSS` 选择 |
| 新闻回退 | `ALLOW_MOCK_FALLBACK` | `true` / `false`                              | `false` | 默认禁用 mock 回退，确保真实新闻源                 |

## 3. 使用说明（当前实现）

| 接口                        | 参数                              | 数据源行为                                                                 |
| --------------------------- | --------------------------------- | -------------------------------------------------------------------------- |
| `GET /api/v1/market/quotes` | `symbols=SPY,QQQ,0700.HK`（可选） | 按 provider 返回标准化行情，字段包含 `source` 与 `updatedAt`               |
| `GET /api/v1/market/stream` | `symbols=SPY,QQQ`（可选）         | SSE 推送标准化行情事件（`type=quotes`），用于实时看板                      |
| `GET /api/v1/news/facts`    | `q=geopolitics&limit=20`（可选）  | 按 provider 返回标准化事实流，字段包含 `source`、`sourceId`、`publishedAt` |

## 4. 设计约束与降级原则

| 规则                       | 落地方式                                          |
| -------------------------- | ------------------------------------------------- |
| 客户端不直接调用第三方 API | 统一通过 NestJS BFF 转发与归一化                  |
| 数据源可替换               | Provider 通过 service adapter 隔离实现细节        |
| 外部失败不阻塞主流程       | 新闻默认走真实外部链路，失败返回可观测错误；可选开启 mock 回退 |
| 数据可追溯                 | 新闻保留 `sourceId`，行情保留 `source` 与更新时间 |
| 去观点与去重               | 新闻在 BFF 层做去观点化清洗、事件 ID 生成与同事件去重 |

## 5. 为什么当前不直接上 AkShare

| 评估项     | 结论                                                           |
| ---------- | -------------------------------------------------------------- |
| 开发复杂度 | 需要新增 Python 服务与网关调用，超出当前“轻量 MVP”范围         |
| 部署复杂度 | 多语言运行时与容器编排更复杂                                   |
| 当前优先级 | 先打通 News/Quote 主链路，再引入 AkShare 作为 Phase 1.5/2 增强 |

## 6. 后续扩展建议

1. 把 `eventId` 去重规则升级为“标题+实体+时间窗口”聚类，减少误并事件。
2. 引入 Redis 对行情和新闻做短 TTL 缓存（P1-03 下一步）。
3. 为每个 provider 增加故障指标与调用成功率统计，纳入 `/api/v1/system/providers` 扩展字段。
